# Premortem: Axolotl Systemic Risks

## 2026-06-17 — Full System Premortem

### Scoring
- **P(робь)**: Probability of failure (Low/Medium/High)
- **S(everity)**: Impact (L1=minor annoyance, L3=partial outage, L5=data loss / full outage)
- **R(isk)**: P × S (L=low, M=medium, H=high, C=critical)

---

## CRITICAL (R=H-C)

### C1. ForkJoinPool thread leak on timeout
**Where**: `NodeRouter.executeNode()` lines 140-240
**Mechanism**: `CompletableFuture.supplyAsync()` dispatches the strategy call into `ForkJoinPool.commonPool()`. When `.get(timeoutSecs, TimeUnit.SECONDS)` at line 220 throws `TimeoutException`, the catch block (line 221-240) sets `FAILED` status, logs, persists error, and `return;` — but **never calls `future.cancel(true)`** or interrupts the strategy thread. The supplyAsync lambda continues running in the common pool.
**Impact**: After N timeouts where N = `Runtime.getRuntime().availableProcessors() - 1` (typically 7-15), the ForkJoinPool is exhausted. No new schema executions can start. The JVM needs restart.
**P**: Medium (1 timeout every ~10 FLUTTER runs on 7B model = plausible)
**S**: L3 (complete execution deadlock until restart)
**R**: H
**Fix**: `.get(timeoutSecs, ...)` → `.completeOnTimeout(...)` or `future.cancel(true)` in catch. Consider dedicated thread pool per schema execution.

### C2. Class-level @Transactional on SchemaExecutionService
**Where**: `SchemaExecutionService.java` line 25 — `@Transactional` on class
**Mechanism**: All public methods including `executeSchema()` (which runs 10+ min pipelines) are wrapped in Neo4j transactions. Neo4j default transaction timeout is 60s. Long-running execution methods will hit `TransactionTimeoutException`, rolling back partially-written state.
**Impact**: Schema execution fails with no way to resume. If the transaction rolls back, in-memory `NodeStatus.RUNNING` may not align with Neo4j state.
**P**: High (every FLUTTER pipeline takes >60s)
**S**: L3 (execution fails, no data loss but UX broken)
**R**: H
**Fix**: Remove `@Transactional` from class, annotate only fast read-only methods (`findExecutionRuns`, `getExecutionHistory` etc.)

## HIGH (R=M-H)

### H1. Gemini provider NullPointerException
**Where**: `LlamaProvider`, `AnthropicProvider`, `OllamaProvider`, `OpenAiProvider` — tool execution paths
**Mechanism**: Agent returns `tool_calls` with `id=null` (observed with some models). Tool execution at `ToolExecutorImpl.java` iterates tool calls without null-guard on `tool_call_id`. Previously fixed once but regression-prone.
**Impact**: Partial agent output is silently lost; execution thread stops with NPE, remaining tools unexecuted.
**P**: Medium (weak models do this regularly)
**S**: L2 (silent partial output — user sees no files generated)
**R**: M-H
**Fix**: Add null-guard at the outer loop level, not per-handler.

### H2. OpenRouter free tier hangs
**Where**: `CustomLlmProvider.sendRawHttpRequestWithRetry()`
**Mechanism**: OpenRouter free-tier proxy does not set Content-Length and keeps HTTP connections open for >120s on combined code-generation requests. The retry mechanism (3 attempts × 120s) means a single planner call can block for 6+ minutes. One hung request blocks a ForkJoinPool thread.
**Impact**: Same as C1 — thread pool exhaustion if multiple concurrent executions hit OpenRouter.
**P**: Medium (every FLUTTER pipeline uses OpenRouter planner)
**S**: L3 (execution stall, possible pool exhaustion)
**R**: M-H
**Fix**: Hard timeout via `CompletableFuture.orTimeout(60s)` on all HTTP calls to OpenRouter (partially done for planner, need for tool agent path too).

### H3. Plugin bridge process zombie leaks
**Where**: `PluginBridge.java` — Bun subprocess lifecycle
**Mechanism**: JVM crash or hard kill (`kill -9`) leaves Bun process running. On restart, a new Bun process starts alongside the zombie. Over multiple restart cycles (deployments, crashes), zombie Bun processes accumulate. Each uses ~50-120MB RSS.
**Impact**: Memory exhaustion on the host machine after 10+ restart cycles.
**P**: Medium (crashes during heavy LLM usage happen)
**S**: L2 (gradual memory pressure, OOM risk)
**R**: M
**Fix**: PluginLifecycleManager should kill orphaned Bun processes at startup (e.g., `pkill -f plugin-bridge.js`). Register JVM shutdown hook.

### H4. Neo4j connection pool exhaustion under load
**Where**: `Neo4jConfig.java`, `application.yml`
**Mechanism**: No explicit connection pool configuration. Spring Data Neo4j defaults may be too small for concurrent schema execution + graph loading + plan persistence. A burst of schema executions could exhaust connections.
**Impact**: `Connection acquisition timeout` — schemas fail to start, graph loading fails silently.
**P**: Low-Medium (depends on concurrent load)
**S**: L3 (all Neo4j operations fail)
**R**: M
**Fix**: Explicit connection pool settings in application.yml.

## MEDIUM (R=M)

### M1. generatedFilesRegistry is in-memory, lost on restart
**Where**: `ExecutionStateManager.java` — `ConcurrentHashMap<String, Object> generatedFilesRegistry`
**Mechanism**: Multi-session workflow relies on `generatedFiles` for cross-session awareness. But the registry is never persisted to Neo4j (the `GraphExecutionRun.setGeneratedFiles()` is never called). Only the `fetchedFromPlan` path in `ProjectContextBuilder` works — and only if PlanService has the data.
**Impact**: After JVM restart, all past session file lists are empty. Agents cannot see what was previously created, violating idempotency rules.
**P**: High (every restart loses file tracking)
**S**: L2 (multi-session development loses context)
**R**: M
**Fix**: Persist `generatedFiles` to Neo4j in `PipelineStageExecutionService.onComplete()`.

### M2. Hardcoded allowedCommands list
**Where**: `ToolHandlerService.java` — `DEFAULT_ALLOWED_COMMANDS` or similar
**Mechanism**: Bash tool command whitelist is a hardcoded list in Java source. Adding a new allowed command requires recompilation and redeployment. No runtime configuration or plugin extension point for command permissions.
**Impact**: User cannot safely add new tools without code change.
**P**: Low (infrequent need)
**S**: L2 (minor friction for power users)
**R**: M
**Fix**: Externalize to application.yml: `axolotl.tools.allowed-commands`.

### M3. WebSocket message buffer overflow
**Where**: `ExecutionWebSocketHandler.java`
**Mechanism**: Agent streaming tokens send individual `sendToken()` messages at high frequency. During long-running agent execution (especially with streaming), the outbound message buffer can grow unbounded. No flow control or backpressure.
**Impact**: WebSocket session memory grows linearly with agent output length. On constrained environments, OOM.
**P**: Medium (long outputs from 14B+ models)
**S**: L2 (memory pressure, possible WS disconnect)
**R**: M
**Fix**: Batch tokens or use fixed-size ring buffer for WebSocket messages.

### M4. No frontend test coverage
**Where**: `frontend/src/` — 0 tests across all Vue components
**Mechanism**: StudioView (889L), QuickStartDialog (811L), TimelineView (880L), DashboardView (1150L), SessionChatDialog (280L) — none have tests. Refactoring these is blind.
**Impact**: Frontend regressions go undetected. Every frontend change is a manual verification risk.
**P**: High (active frontend development)
**S**: L1 (regressions caught late, slow dev cycle)
**R**: M
**Fix**: Add Vitest for composables at minimum (`useExecution`, `useReview`).

## LOW (R=L)

### L1. Multiple schemaRepository.findById() calls within same method
**Where**: Several strategies call `schemaRepository.findById(schemaId)` multiple times in the same method (e.g., AgentNodeStrategy before the recent fix had 2 calls). Each call is a separate Neo4j query.
**Impact**: 2-4 redundant Neo4j queries per node execution. Adds ~20-40ms overhead.
**P**: High (always happens)
**S**: L1 (minor latency)
**Fix**: Cache locally at method start.

### L2. Pipeline retry duplicates stage execution history
**Where**: `PipelineStageExecutionService.retryPipeline()`
**Mechanism**: Retrying a pipeline creates new ExecutionRun, but the previous run's stage outputs remain visible in the UI. User sees duplicate "Session N" entries.
**Impact**: Timeline gets cluttered with failed+retry pairs.
**P**: High (retry is a common workflow)
**S**: L1 (cosmetic)
**Fix**: Hide retry child runs from timeline, or merge into parent.

### L3. No LLM cost tracking per-user
**Impact**: Cannot bill per tenant.
**P**: Low
**S**: L1
**Fix**: Add tenant ID to LlmUsage record.

## Summary

| Severity | Count | Key items |
|----------|-------|-----------|
| CRITICAL | 2 | Thread leak on timeout, @Transactional on long methods |
| HIGH | 4 | Null tool id, OpenRouter hangs, zombie Bun, Neo4j pool |
| MEDIUM | 4 | File registry lost on restart, hardcoded commands, WS buffer, no frontend tests |
| LOW | 3 | Redundant DB calls, retry clutter, cost tracking |

**Top 3 fixes by ROI**:
1. **`future.cancel(true)` in NodeRouter timeout catch** — removes thread leak, prevents pool exhaustion
2. **Remove class-level `@Transactional` from SchemaExecutionService** — prevents Neo4j timeout failures on every pipeline run
3. **Persist `generatedFiles` to Neo4j** — makes multi-session actually survive restart
