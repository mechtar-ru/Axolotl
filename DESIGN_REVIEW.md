# Посмертный дизайн-аудит Axolotl

> Ретроспективный анализ причин провала продукта через призму UX/UI дизайна.
> Дата анализа: 2026-05-10 (6 месяцев после релиза).

---

## 1. Когнитивный обрыв: 14 типов узлов и никаких ориентиров

Пользователь заходит. OnboardingModal — 2 шага: выбрать провайдер → "готово". Это не онбординг, это чекбокс. Никакого объяснения, чем `condition` отличается от `guardrail`, почему `fallback` — отдельный тип, а `subagent` не тот же `agent`. 14 типов узлов без визуальной иерархии — пользователь видит выпадающее меню из 12 пунктов, сгруппированных в «Данные / Агенты / Логика / Интеграция / Утилиты», но категории не помогают: `memory` — это данные? интеграция? `command` — логика?

> **Принцип когнитивной нагрузки Миллера**: 7±2 элемента в рабочей памяти. 14 узлов — вдвое больше. Без визуального гида или progressive disclosure мозг пользователя переключается в режим сканирования вместо понимания.

Было 3 шаблона на empty state, но они не раскрывали сложность: `AI Pipeline` — линейный, `RAG` — линейный, `Condition Branch` — единственный с ветвлением. Ни один не показывает `loop`, `transform` с routes, `fallback`, `schemabuilder`. Пользователь не узнаёт, что продукт умеет, пока не уткнётся в выпадающее меню.

**Файлы**: `OnboardingModal.vue`, `HomeView.vue:61-81` (template cards), `WorkflowCanvas.vue:44-77` (add dropdown)

## 2. Раскрытие узла = раскопки

Каждый узел по умолчанию свёрнут. Чтобы настроить `AgentNode`, нужно: клик → нажать `▶` → увидеть 5-строчный textarea, select модели, секцию инструментов (тоже свёрнутую!), лимит вызовов. AgentNode — самый частый узел, но его конфигурация — 3 уровня вложенности (expand node → expand tools → select type + check tools).

`LoopNode` и `ConditionNode` — текстовое поле «условие» без подсказок, какой синтаксис ожидается. Placeholder: `iterations < 10` для loop, `data.value > 10` для condition. Но нигде не сказано, что condition выполняется через GraalVM, какие переменные доступны, что такое `data`. Пользователь пишет `input.length > 50` (из шаблона!), не понимая, откуда берётся `input`.

> **Affordance (Дональд Норман)**: интерфейс должен показывать, *что* с ним можно делать. Свёрнутый узел с кнопкой `▶` говорит «здесь что-то есть», но не говорит *что*. Нет preview промпта, нет badge с количеством инструментов, нет иконки статуса конфигурации.

**Файлы**: `AgentNode.vue:39-94`, `LoopNode.vue:36-61`, `ConditionNode.vue:40-55`

## 3. Двойная жизнь данных: `data` vs `config`

SourceNode хранит данные в `sourceData`, но конфигурацию — в `config.sourceType`, `config.projectPath`, `config.memoryWing`. OutputNode — `outputType` в корне data, но `filePath` тоже в корне, а `memoryWing` — в `config`. TransformNode — `transforms`, `routes`, `fallbackValue` — всё в корне data.

Пользователь не видит эту путаницу напрямую, но она проявляется в багах: при копировании узла `data` копируется поверхностно, `config` может теряться. При импорте/экспорте JSON структура непрозрачна — невозможно понять, какие поля важны.

**Файлы**: `SourceNode.vue:144-158` (data props), `OutputNode.vue:104-120` (data props), `WorkflowCanvas.vue:786-815` (addNode defaults)

## 4. Нет обратной связи при настройке

`TransformNode` — самый сложный узел. Пользователь должен: выбрать тип transform из 15 опций (jsonField, jsonPath, regex, delimited, replace, prepend, append, lower, upper, trim, template, jsonParse, jsonStringify, ifEmpty), ввести параметры, затем настроить Routes — где нужно вручную вписать `targetNodeId` (строка ID!). Нет способа выбрать узел из списка. Нет валидации. Нет preview. При выполнении пользователь видит «результат», но не видит, *какой* transform сработал, какой route выбран, почему.

> **Principle of Visibility (Нильсен)**: пользователь должен видеть состояние системы. В TransformNode routes ссылаются на `targetNodeId` как raw string — нет autocomplete, нет визуальной связи с canvas. Это скрытая coupling: изменение ID узла ломает route молча.

**Файлы**: `TransformNode.vue:40-158`

## 5. Execution Panel — хроника смерти

При выполнении открывается правая панель. Логи — плоский список timestamped строк. Волны — номера без имён узлов (`Волна 1: 3 узлов`). Траектория — итерации и tool calls, но без содержимого: только имя инструмента, duration, success/fail. Нет *результата* инструмента. Пользователь не может понять, что агент делал между итерациями.

Progress bar — линейный процент. Но execution — wave-based parallel. Три узла в волне → 33% → 66% → 100%. Если один из трёх падает, пользователь видит 66% и... не видит, какой упал. Нужно кликать по логам, надеясь найти строку с `error`.

**Файлы**: `ExecutionPanel.vue:1-121`, `WorkflowCanvas.vue:853-961` (execute flow)

## 6. Правая панель: швейцарский нож без рукоятки

5 иконок-табов в правой панели (▶️, 📋, 🧠, 📜, 🏗️) без лейблов — только tooltips. Exec, Plan, Memory, History, Templates — пять разных сущностей в одном контейнере. Tab icons — 16px эмодзи, различить которые на Retina-экране — задача. Нет визуальной индикации, что в History есть данные, в Memory — нет, Templates — пустые.

Панель resizable через drag handle слева, но нет кнопки collapse/expand, нет минимума — можно сжать до 0px. На мобильном панель перекрывает canvas полностью.

**Файлы**: `RightPanel.vue:1-115`, `panelStore.ts`

## 7. Билингвальный хаос

UI — смесь русского и английского. Кнопки: «Выполнить», «Сохранить», «Экспорт» — рядом с «Execute», «Analyze», «Dry Run» (mode selector). Node labels: «Входные данные» (source), «Аналитик» (agent) — но Node Types в dropdown: «Source», «Agent», «Output». Шаблоны: «AI Pipeline», «RAG Pipeline», «Condition Branch» — но «Демо-схема». Toolbar: «Добавить», «Группа», «Разгрупп.» — но «Memory», «Templates», «Plan». Settings: «Модель по умолчанию» — но «Custom LLM APIs», «Add new», «Enabled».

> **Linguistic consistency** — не перфекционизм, а снижение когнитивной нагрузки. Переключение языков заставляет мозг делать context switch. Для непрограммиста «Разгрупп.» и «Ungroup» — два разных понятия.

**Файлы**: `WorkflowCanvas.vue:16-22` (mode selector), `WorkflowCanvas.vue:46-77` (dropdown), `SettingsView.vue:108-199` (custom endpoints), `HomeView.vue:61-81` (templates)

## 8. Schema Builder — обещание без доставки

Empty state показывает SchemaBuilder как выделенную (жёлтый border) action card с текстом «Опиши приложение — AI создаст схему». Это главная точка входа для нового пользователя, самая привлекательная функция. Но результат — сырая схема с пустыми промптами, без объяснения, что делать дальше. Пользователь получает 5 узлов, соединённых линиями, и... должен сам разобраться, что писать в каждый. Нет guided flow после генерации.

**Файлы**: `HomeView.vue:100-104` (SchemaBuilder card), `SchemaBuilderModal.vue`

---

## 9. Никакой learnability

В продукте 30+ keyboard shortcuts (Ctrl+S, Ctrl+Z, Ctrl+F, Ctrl+Enter, Ctrl+E, Ctrl+G, Ctrl+D, ?, Escape), но они нигде не показаны, кроме ShortcutsOverlay (который открывается по `?`, о чём никто не знает). Нет tooltip'ов на кнопках (большинство — просто эмодзи без text label). Нет guided tour после онбординга. Нет contextual help.

В `NodeContextMenu` (правый клик) — rename, duplicate, delete, collapse, edit prompt. Но rename вызывает `prompt('Новое имя:')` — нативный browser dialog. В 2026 году. Для продукта, который хочет быть визуальным.

**Файлы**: `WorkflowCanvas.vue:622-724` (keyboard handler), `NodeContextMenu.vue`, `ShortcutsOverlay.vue`

---

## 10. Settings как кладбище

Настройки провайдеров — отдельная страница, покинув canvas. Нет inline-настройки: пользователь видит, что агент не работает, но должен: выйти на Home → Settings → найти провайдер → ввести ключ → тест → вернуться. Нет hint'а на canvas «провайдер не настроен» — только `console.warn` в консоли.

**Файлы**: `SettingsView.vue`, `WorkflowCanvas.vue:308-324` (model loading with `console.warn`)

## Резюме: death by a thousand papercuts

Axolotl умер не потому, что backend медленный или LLM плохой. Он умер потому, что **никто не смог понять, как им пользоваться**. Продукт требовал от пользователя:

1. Знать, что 14 типов узлов существуют и чем отличаются
2. Понимать модель данных (data/config/result/status) без документации
3. Угадывать синтаксис условий без автодополнения
4. Настроить провайдера на отдельной странице до первого запуска
5. Читать плоские логи для понимания, что пошло не так
6. Знать keyboard shortcuts, нигде не подсвеченные

Каждый шаг в отдельности — преодолим. Но вместе они образовали **death by a thousand papercuts**: пользователь тыкался 10 минут, не мог заставить схему выполниться, закрывал таб и не возвращался.

> **Закон Фиттса в UX**: время достижения цели пропорционально дистанции и обратно пропорционально размеру цели. В Axolotl дистанция (шаги настройки) велика, а цель (успешное выполнение) — мала и невидима. Каждая лишняя модалка, каждый свёрнутый узел, каждый непонятный placeholder — увеличение дистанции без увеличения цели.
