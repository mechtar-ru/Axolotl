Requirements Specification: «Как мы чувствуем» (Emotion Tracker)
1. Introduction
1.1 Purpose

The application helps Russian‑speaking users identify, log, and understand their emotions. It provides a culturally adapted emotional vocabulary, visual analytics, and actionable self‑care tips.
1.2 Scope

    Native or cross‑platform mobile app (iOS & Android).

    Offline‑first with optional cloud sync.

    Supports individual use only (no social features).

1.3 Target Audience

    Russian‑speaking adults (18+) interested in emotional self‑tracking.

    People with no clinical diagnosis, but may include users with mild anxiety or mood fluctuations.

2. User Stories (summary)
ID	As a…	I want to…	So that…
US‑01	User	Log my current emotion in ≤10 seconds	I build a consistent tracking habit
US‑02	User	See a calendar with colour‑coded emotions	I quickly recognise weekly/monthly patterns
US‑03	User	Add a text note to an emotion entry	I remember what triggered the feeling
US‑04	User	Receive a tip relevant to my logged emotion	I learn a regulation strategy immediately
US‑05	User	Export my data (JSON/CSV)	I can share with a therapist or analyse myself
US‑06	User	Delete all my data (local + cloud)	I exercise my right to privacy (152‑FZ)
3. Functional Requirements
3.1 Onboarding & Authentication

| FR‑01 | App shows a short intro (2 screens) explaining the value and privacy policy. |
| FR‑02 | Guest mode: use without registration – data stored only locally (device). |
| FR‑03 | Full account: registration with email + password, or via Google / VK ID / Apple ID. |
| FR‑04 | User can upgrade from guest to full account (data merged). |
| FR‑05 | Account deletion permanently removes all entries from device and cloud (if any). |
3.2 Emotion Logging (Core Feature)

| FR‑06 | Main screen displays a colour‑based emotion wheel/grid with 6 basic categories (see 3.2.1). |
| FR‑07 | Each category expands into a detailed list of ≥12 specific feelings (Russian vocabulary – see Appendix A). |
| FR‑08 | User selects intensity (1 = very weak, 5 = very strong) via slider or 5 buttons. |
| FR‑09 | Optional fields: text note (≤500 chars), context tags (Home/Work/Public/Alone), activity (walking, working, etc.). |
| FR‑10 | Timestamp recorded automatically (user can edit date/time up to 3 days back). |
| FR‑11 | After saving, user returns to main screen; a subtle confirmation animation appears. |
| FR‑12 | Minimum one emotion entry per day is encouraged via optional push reminder (configurable time). |

3.3 History & Analytics

| FR‑13 | Calendar view (month grid) – each day coloured by dominant emotion (if multiple entries – average or most frequent). |
| FR‑14 | Tap a day → list of entries for that day (time, emotion, intensity, note preview). |
| FR‑15 | Statistics screen shows: |
| | – Top 5 most frequent specific feelings (last 30 days). |
| | – Emotion distribution as a donut chart (by basic colour). |
| | – Intensity average per basic emotion. |
| | – Line graph: mood trend (averaged intensity over time) by week/month/year. |
| FR‑16 | User can filter analytics by date range (presets: 1W, 1M, 3M, 1Y, custom). |

3.5 Settings

| FR‑21 | Dark / Light theme (follow system or manual override). |
| FR‑22 | Notification settings: daily reminder, weekly summary. |
| FR‑23 | Export data. |

4. Non‑Functional Requirements
4.1 Performance & Offline

| NFR‑01 | App launches within 2 seconds on a mid‑range device (Android 10+ / iOS 14+). |
| NFR‑02 | Emotion log saves in <0.3 seconds (local DB). |
| NFR‑03 | Works fully offline – all features except sync. |
| NFR‑04 | When online, sync happens in background without blocking UI. Conflict resolution: last write wins (timestamps). |
4.2 Usability

| NFR‑05 | Max 3 taps to log an emotion (starting from home screen). |
| NFR‑06 | Font size respects system accessibility settings (Dynamic Type on iOS, font scaling on Android). |
| NFR‑07 | Supports VoiceOver (iOS) and TalkBack (Android) with proper labels for all interactive elements. |
4.3 Privacy & Legal (Russian 152‑FZ)

| NFR‑08 | Before first data collection, display consent screen with: |
| | – What data is collected (emotions, notes, context, device ID for notifications). |
| | – Purpose (personal tracking, no third‑party sharing). |
| | – Storage location (user’s device + optional cloud if user registers). |
| NFR‑09 | User can delete all data (account deletion) without contacting support. |
| NFR‑10 | No analytics or crash reporting tools that send data outside Russian jurisdiction unless explicitly approved and disclosed. (Prefer open‑source or self‑hosted.) |
4.4 Security

| NFR‑11 | If cloud sync used: all data encrypted in transit (TLS 1.3) and at rest (AES‑256). |
| NFR‑12 | Authentication tokens stored securely (iOS Keychain / Android Keystore). |
| NFR‑13 | No plaintext logging of notes or emotions in system logs. |
5. Data Models
5.1 Local Database Schema (SQLite / Room / Core Data)

Entity EmotionEntry
Field	Type	Description
id	UUID / string	Primary key
timestamp	ISO8601 (UTC)	When emotion was felt (user‑selected or now)
basic_emotion	enum (6 values)	Green, Blue, Red, Yellow, Purple, Grey
specific_feeling	string (max 50 chars)	e.g., «тоска»
intensity	integer (1–5)	
note	text (optional)	≤500 chars
context_tags	JSON array of strings	e.g., ["home", "work"] – max 3 tags
activity	string (optional)	free text
synced	boolean	0 = not synced, 1 = synced (if cloud used)

Entity UserSettings (key‑value)

    daily_reminder_enabled (bool)

    daily_reminder_time (string “HH:MM”)

    theme (string: light / dark / system)

    user_id (if authenticated) – optional

5.2 Cloud Sync (optional – Firestore / custom backend)

Same EmotionEntry structure plus:

    user_id (foreign key)

    updated_at (server timestamp for conflict resolution)

6. UI/UX Requirements
6.1 Screens & Navigation

Tab bar (bottom navigation) with 4 tabs:

    Log – emotion grid + quick log button (if no emotion selected)

    History – calendar view + list of entries for selected day

    Analytics – charts + statistics + export button

    Profile – settings, resources, account management

6.2 Wireframe descriptions (text)

    Log Screen: at the top – current date and time (editable pencil icon). Below – 6 large colour circles arranged in 2 rows of 3. Each circle contains emotion name (Спокойствие, Печаль, etc.) in white text. Tapping a circle → bottom sheet slides up showing detailed list of specific feelings. After selecting feeling → intensity slider appears, then optional note field. “Save” button at bottom.

    History Calendar: month header with arrows. Days with entries show small colour dot (dominant basic emotion). No dot = no entry. Tap day → list of entries (time, emotion, intensity). Swipe left on an entry to delete.

    Analytics: default view shows last 30 days. Line chart (intensity over time). Below that, donut chart (emotion distribution). Top 5 list. A “share” icon triggers export.

6.3 Russian localisation specifics

    Date format: DD.MM.YYYY

    Time format: 24‑часовой (e.g., 14:30)

    Emotion names must be grammatically correct in the context of “Я чувствую …” (nominative case). Example: «Я чувствую тоску» not «тоска».

    Tips use informal “ты” (since the app is personal).

7. Technical Stack Recommendation
Layer	Recommended choice	Justification
Cross‑platform	Flutter (Dart)	Single codebase, excellent performance, strong SQLite support, Russian‑friendly community. React Native is acceptable but Flutter offers more consistent UI on both platforms.
Local DB	SQLite via sqflite (Flutter)	Reliable, ACID, works offline.
State management	Riverpod or BLoC	Predictable, testable.
Optional cloud sync	Firebase (Firestore + Auth)	Quick to implement, but must inform users about data location. Alternative: Supabase (PostgreSQL) for more control.
Push notifications	Firebase Cloud Messaging (FCM) / Apple Push Notification Service (APNS) via cloud functions	Standard.
Analytics (optional)	Self‑hosted Plausible or none (to comply with 152‑FZ)	Avoid sending emotion data to third‑party analytics.
8. Constraints & Assumptions

    No real‑time collaboration or sharing.

    No clinical diagnosis or emergency intervention – the resources screen includes a disclaimer: «Приложение не заменяет профессиональную помощь. При кризисной ситуации звоните 112 или на горячую линию.»

    Initial version supports Russian language only.

    Minimum supported OS: iOS 14, Android 10 (API level 29).

9. Acceptance Criteria (sample)

| AC‑01 | User can log an emotion from cold start in ≤10 seconds (measured on iPhone 11 / Pixel 4a). |
| AC‑02 | Calendar shows correct colour coding for days with multiple entries (dominant colour). |
| AC‑03 | Export produces a valid JSON file containing all entries. |
| AC‑04 | After deleting account, reopening the app shows onboarding screen (no leftover data). |
| AC‑05 | App works without internet – all logs are saved locally and appear in history. |
Appendix A – Russian specific feelings (non‑exhaustive)

Green (Спокойствие)
умиротворение, безмятежность, интерес, надежда, принятие, доверие, расслабленность

Blue (Печаль)
тоска, грусть, разочарование, одиночество, ностальгия, безнадёжность, уныние

Red (Гнев)
раздражение, злость, ярость, презрение, негодование, обида, ревность

Yellow (Тревога)
беспокойство, страх, паника, нервозность, неуверенность, стыд, вина

Purple (Удовольствие)
восторг, радость, гордость, вдохновение, нежность, умиление, благодарность

Grey (Апатия)
усталость, безразличие, опустошение, скука, отупение, оцепенение

    Each term must be displayed in the nominative case, but inserted into the sentence “Я чувствую…” as the accusative (e.g., «Я чувствую тоску»). The UI will show: ТОСКА (header) and description “Я чувствую тоску”.
